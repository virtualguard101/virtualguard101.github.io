Summary

参与了11月23号在福州举办的TRAE Friends活动，对活动分享的内容做一个简要记录

为什么不做CLI产品

即类似Claude Code、Codex等的CLI Agent

CLI -> SOLO
(vi) -> IDE

CLI并不是产品的最终形态，IDE才是

如何高效进行VibeCoding

工具只是载体，VibeCoding本身是工具无关、模型无关的。

重要的是在开发过程中如何沉淀出自己的工作流与方法论。

软件开发的“旧”常态

  • 繁琐的模板代码

  • 语法的细枝末节

  • 创意与实现之间的鸿沟

AI Coding现状的破局

本身并不是一个“新”问题

CTO如何协调掌握各个领域的专业知识，协调各个部门的工作?

将VideCoding工具看作“能力很强的”实习生，如何去高效地指挥与协调?

  • 从 “编码者” 到 “指挥者” 的转变

    • 项目构想

    • 需求拆解

    • 结果验证

    • 审美决策

VibeCoding 的核心心法

先求成，再求精 (Code First, Refine Later)

  • 接受不完美

  • 拥抱迭代

  • 降低试错成本

高效实践工作流

  • 编写一份高质量的提示词

    • 文档驱动开发 (Documentation-Driven Development)

    • 产品经理规划 (Vibe PMing)

    • 多模态输入 (视觉辅助等)

  • 上下文精准注入与原子化

    • 一次对话，实现一个功能

      拆分任务 -> 精准提问 -> AI实现 -> 人类验证 -> 优化迭代

  • 将trae作为一个"调试助手"

    • 测试驱动开发 (Test-Driven Development)

    • 错误信息 “投喂”

    • 代码 “体检” 与重构

高效VibeCoding的核心原则

  • 原则一: 指令清晰高于一切 (Clarity First)

  • 原则二: 小步快跑，持续验证 (Iterative Control)

  • 原则三: 人类监督，绝不缺位 (Human Oversight)

  • 原则四: 拥抱学习，共同成长 (Learning on the Go)

杂项

  • 前后端交互的最佳方式是API文档

  • 在AI时代，得益于AI工具对信息差降低乃至抹平，人在不同行业间的流动性将获得极大提升
